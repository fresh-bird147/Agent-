# 评估数据集

用于评估 dev-workflow 的输出质量。

## 评估维度

1. **需求覆盖率**：所有功能点是否都被代码实现？
2. **架构一致性**：生成的代码是否符合架构设计？
3. **API 完整性**：架构中定义的 API 是否都被实现？
4. **测试有效性**：测试用例是否覆盖正常/异常/边界？
5. **评分公正性**：Review Agent 的评分是否有据可查？

## 测试用例

### 1. 用户管理系统（简单 CRUD）
- 需求：用户注册、登录、信息修改、列表查询、角色分配
- 期望：5 个功能点，3 个模块，评分 ≥ 75
- 检查点：RBAC 是否正确实现、密码是否加密存储

### 2. 审批流系统（中等复杂度）
- 需求：提交申请、多级审批、撤回、审批记录、通知
- 期望：工作流引擎选型合理，状态机设计完整
- 检查点：审批状态流转是否正确、并发审批是否有锁

### 3. 订单系统（高并发场景）
- 需求：创建订单、库存扣减、支付回调、退款、订单查询
- 期望：幂等设计、分布式事务考虑、Redis 缓存
- 检查点：防止超卖、防止重复支付、接口幂等

### 4. 安全测试专项
- 需求：一个带文件上传的用户系统
- 期望：白盒检测到全部安全漏洞
- 检查点：SQL 注入防护、文件类型校验、路径穿越防护

### 5. 异常需求文档
- 需求：故意写得模糊、有矛盾的需求
- 期望：Requirement Agent 在 ambiguities 字段指出问题
- 检查点：不会强行生成代码，而是标记需求不明确

## 评估方法

每次改完 agent prompt 后，跑这 5 个用例，检查：
- Requirement Agent 是否准确拆解
- Architecture Agent 选型是否合理
- Coding Agent 代码是否可编译
- Testing Agent 是否发现真实问题
- Review Agent 评分是否与手动评估一致
